\begin{abstract}
We present the Temporal Gait Transformer (TGT), a classification model that takes raw smart insole sensor data and identifies five distinct walking activities: normal walking, injury walking, stepping, swaying, and jumping. The input data comes from pressure sensors, accelerometers, and orientation quaternions on both feet. After merging left and right insole streams into synchronised rows at \SI{60}{Hz}, we segment the recording into overlapping windows of 80 timesteps (1.33 seconds, roughly one full gait cycle). Each timestep carries 50 sensor features. Every window is independently z-score normalised per feature to remove subject-specific sensor baselines. The TGT uses a two-layer 1D convolutional stem to extract local temporal patterns such as heel-strike and toe-off phases, then passes the result through two Transformer encoder layers with multi-head self-attention to capture longer-range gait dynamics. The output is obtained by mean-pooling across timesteps rather than a learnable classification token, which we found more stable given limited training subjects. The model is trained with label-smoothed weighted cross-entropy loss and the AdamW optimiser with cosine annealing. Aggressive data augmentation (Gaussian jitter, per-channel scaling, channel dropout, temporal masking, and random time shifts) is applied during training to simulate cross-subject variability and reduce overfitting. All data splits are performed at the file level so that overlapping windows from the same recording never appear in both training and validation sets.
\end{abstract}

\section{Data Preprocessing}

\subsection{Data Types}
Data is collected from smart insoles on both feet of each subject. At every time step, the insoles measure pressure at multiple locations and provide accelerometer, gyroscope, and magnetometer readings, along with a computed quaternion and an insole label (right or left). Redundant fields include internal time stamps for each sub-device and battery status. A detailed list of all per-timestep features is provided in Table~\ref{tab:insole-features} in the Appendix.

\subsection{Separating Input Data}
The raw data for each walking condition is first separated into right and left insole streams using the insole identifier $ele\_36$. We then compare the time stamps of rows from the right and left insoles and match them into a single combined row when the difference in recording times satisfies the constraint $\delta t < 0.02$\,s. Rows that do not satisfy this constraint are discarded. We remove redundant sub-time fields and drop gyroscope and magnetometer readings, because this information is already captured by the quaternions. The resulting dataset, with all retained channels, is summarised in Table~\ref{tab:insole-features-selected} in the Appendix. Each remaining row is a full entry for both feet, containing 50 features in total: 18 pressure values, 3 accelerometer axes, and 4 quaternion components per foot.

\subsection{Sampling Rate}
After merging, the data is resampled to a fixed \SI{60}{Hz}. If the merged stream has a frame rate below 60\,Hz we upsample by linear interpolation. If it is above 60\,Hz we downsample by selecting evenly spaced indices. This ensures that every recording, regardless of the original device clock, produces a uniform 60 samples per second stream.

\paragraph{Note on upsampling correction.}
An earlier version of the upsampling routine generated evenly spaced target timestamps and called \texttt{reindex} directly on the original data frame. Because the target timestamps almost never coincided exactly with the recorded timestamps (floating-point mismatch on ten-digit Unix epochs), the operation discarded virtually all original samples and left a single anchor point, causing \texttt{interpolate} to replicate one constant row across the entire recording. This affected 35 of 92 Injury\_walking training files and all four Group~10 recordings, whose native frame rate fell below \SI{60}{Hz}. The corrected implementation first takes the union of the original and target timestamp indices, interpolates over the combined index so that every original measurement serves as an anchor, and then selects the target timestamps. After this fix, all upsampled recordings contain the expected temporal variation.

\subsection{Windowing}
We define a window parameter $W$ as the number of consecutive rows provided to the model. In our configuration we use $W = 80$, which corresponds to 1.33 seconds at \SI{60}{Hz}, enough to cover a full gait cycle with some margin. The stride between consecutive windows is set to $S = 20$, giving 75\% overlap. This overlap produces more training samples and ensures that transient gait events are captured even if they fall near a window boundary. Each window is therefore a matrix of shape $80 \times 50$.

\subsection{Normalisation}
We apply two layers of normalisation. First, a global z-score is computed per feature across the entire training set and applied to all data (training, validation, and evaluation). The mean and standard deviation are fit only on training windows to prevent information leakage. This brings all features onto a comparable scale.

Second, each window is independently z-score normalised per feature. For each window $\mathbf{X} \in \mathbb{R}^{W \times F}$, we compute the mean $\mu_f$ and standard deviation $\sigma_f$ of each feature $f$ across the $W$ timesteps and normalise:
\begin{equation}
\hat{x}_{t,f} = \frac{x_{t,f} - \mu_f}{\sigma_f + \varepsilon}
\end{equation}
where $\varepsilon = 10^{-8}$. This removes subject-specific absolute sensor baselines (e.g.\ resting pressure levels and accelerometer offsets that vary with body weight, foot placement, and device calibration) so that the model learns from the temporal \emph{shape} of each window rather than its absolute magnitude.

\subsection{Split}
Because we use overlapping sliding windows, a naive random split would place near-identical windows into both the training and validation sets, inflating validation scores. To avoid this data leakage, we split at the \emph{file} level: all windows from the same CSV recording are assigned to the same partition. We use a 90/10 train/validation split. The validation set is used during training to monitor generalisation and select the best checkpoint via early stopping. All available data, including the Group~10 recordings, is included in the training pool to maximise subject diversity. Final evaluation is performed on a completely separate set of recordings not seen during training.


\section{Methodology}

The TGT is a hybrid CNN-Transformer model. A 1D convolutional stem first extracts local temporal features from the raw sensor window, then a Transformer encoder captures longer-range dependencies across the gait cycle. The output is produced by mean-pooling across all timesteps rather than a learnable classification token, which we found more reliable when training data comes from only a handful of subjects. We describe each stage below.

\subsection{1D Convolutional Stem}

The input to the model is a window of shape $W \times 50$, where each of the $W = 80$ rows is a single timestep with 50 sensor features. Rather than projecting each timestep independently through a linear layer, we first transpose the input to shape $50 \times W$ and apply two 1D convolutional layers along the time axis.

The first convolution uses a kernel size of 7 and maps the 50 input channels to $d_{\text{model}} = 64$ output channels. The second convolution uses a kernel size of 5 and maps 64 channels to 64. Both use same-padding so the temporal length stays at $W = 80$. Each convolution is followed by batch normalisation, a GELU activation, and dropout.

The motivation for a convolutional stem is that gait signals contain local temporal patterns (the pressure wave from heel-strike to toe-off, the acceleration peak at foot contact) that are best captured by filters operating over small neighbourhoods. These local features tend to be more consistent across subjects than the raw sensor magnitudes, which vary with body weight, sensor placement, and foot shape. After the stem, we transpose back to shape $W \times 64$ for input to the Transformer.

\subsection{Positional Encoding}

Since the Transformer has no built-in notion of ordering, we add a fixed positional encoding to each token so the model can distinguish early timesteps from later ones within the window. For a given position $\text{pos} \in \{0, 1, \dots, W{-}1\}$ and a dimension index $i \in \{0, 1, \dots, 31\}$ we compute:

\begin{align}
PE_{\text{pos},\, 2i}   &= \sin\!\left(\frac{\text{pos}}{10000^{\,2i / d_{\text{model}}}}\right) \\[4pt]
PE_{\text{pos},\, 2i+1} &= \cos\!\left(\frac{\text{pos}}{10000^{\,2i / d_{\text{model}}}}\right)
\end{align}

with $d_{\text{model}} = 64$. Lower dimensions oscillate rapidly and capture fine-grained local timing, while higher dimensions oscillate slowly and encode where the timestep sits within the segment. The result is a $80 \times 64$ matrix of positional vectors added element-wise to the convolutional features.

\subsection{Transformer Encoder}

The core of the model is a stack of $N = 2$ identical Transformer encoder layers. Each layer consists of two sub-blocks.

\subsubsection{Multi-Head Self-Attention}

For each of the 80 tokens, the layer computes Query ($Q$), Key ($K$), and Value ($V$) vectors via learned linear projections. The attention score between any two tokens is the scaled dot product of one token's query with another's key. These scores form an $80 \times 80$ matrix, passed through a softmax so that each row sums to one. Multiplying by the Value matrix produces a new representation where each token is a weighted combination of all tokens, according to their relevance to it.

We use $H = 4$ attention heads in parallel. Each head operates on a $64/4 = 16$ dimensional slice of the embedding, and the results are concatenated back to 64 dimensions.

\subsubsection{Feed-Forward Network}

After attention, each token passes independently through a two-layer feed-forward network (FFN). The FFN projects from 64 to 256 dimensions (a 4$\times$ expansion), applies a GELU activation, and projects back to 64 dimensions. Residual connections and layer normalisation are applied around both the attention sub-block and the FFN sub-block.

A dropout rate of 0.3 is applied inside both the attention and FFN sub-blocks. The two encoder layers are applied sequentially, with each one refining the token representations further.

\subsection{Mean Pooling and Classification Head}

After the final encoder layer, we compute the mean of all 80 token representations along the time axis, producing a single 64-dimensional vector. We chose mean pooling over a learnable [CLS] token because, with training data from only a small number of subjects, the CLS token tended to overfit to subject-specific patterns rather than learning a useful aggregate representation. Mean pooling is parameter-free and gives a stable summary regardless of training set size.

This pooled vector is passed through layer normalisation, dropout, and a fully connected linear layer that outputs 5 values, one per class. A softmax is applied to these logits to obtain a probability distribution over the five activities.

\subsection{Training Configuration}

The model is trained with backpropagation using cross-entropy loss with label smoothing ($\alpha = 0.1$) between the predicted class probabilities and the ground-truth labels. Label smoothing prevents the model from becoming overconfident on the training set, which is especially useful when the number of distinct subjects is small.

Because the training set is class-imbalanced (e.g.\ Swaying has roughly twice as many windows as Stepping), we address this at two levels. First, we weight the loss by the inverse class frequency so that under-represented activities receive proportionally higher gradients. Second, we use a weighted random sampler during training: each window is sampled with probability inversely proportional to the frequency of its class, so that every class is seen roughly equally often per epoch regardless of the original distribution. This combination of loss weighting and oversampling ensures that minority classes like Stepping receive sufficient training signal despite having far fewer recording sessions.

We use the AdamW optimiser with decoupled weight decay. Gradients are clipped to a maximum norm of 1.0 per step to prevent occasional large updates from destabilising training. The specific training parameters are listed in Table~\ref{tab:training-params}.

\subsection{Data Augmentation}

Because training data comes from a small number of subjects, we apply aggressive augmentation to each training window on every epoch to simulate the kind of variation expected from new subjects. The five transforms are:

\begin{enumerate}
\item \textbf{Gaussian jitter.} Additive noise sampled from $\mathcal{N}(0, 0.15^2)$ is added independently to every element of the window. This simulates sensor noise variation between recording sessions and devices.
\item \textbf{Per-channel amplitude scaling.} Each of the 50 features is multiplied by an independent random factor drawn uniformly from $[0.5, 1.5]$. This forces the model to tolerate differences in sensor calibration and body weight across subjects.
\item \textbf{Channel dropout.} Each feature channel is independently zeroed out with probability 0.2. This prevents the model from relying too heavily on any single sensor and improves robustness to occasional sensor failures.
\item \textbf{Temporal masking.} A contiguous block of up to $\lfloor W/7 \rfloor$ timesteps (roughly 15\% of the window) is randomly selected and set to zero, acting as a regulariser that forces the model to use the full temporal context.
\item \textbf{Random time shift.} The window is circularly shifted along the time axis by a random offset of up to $\pm W/8$ timesteps. This makes the model invariant to where exactly within the recording the gait event starts.
\end{enumerate}

Augmentation is applied online during training only; validation and evaluation data are not augmented.

In addition to these per-window transforms, we apply \textbf{mixup} at the batch level during training. For each mini-batch, we sample a mixing coefficient $\lambda \sim \text{Beta}(\alpha, \alpha)$ with $\alpha = 0.2$ and blend each input window with a randomly paired window from the same batch: $\tilde{x} = \lambda\, x_i + (1 - \lambda)\, x_j$. The loss is computed as $\lambda\, \mathcal{L}(f(\tilde{x}),\, y_i) + (1 - \lambda)\, \mathcal{L}(f(\tilde{x}),\, y_j)$. With $\alpha = 0.2$ the mixing is usually mild (most $\lambda$ values are close to 0 or 1), but it still smooths decision boundaries and reduces the tendency to memorise individual training windows.
